# 项目体验优化建议

> 评估时间：2026-04-17  
> 评估范围：Web 学生端、Web 教师端、管理端、微信小程序端、核心 bootstrap 数据链路  
> 原则：**不新增功能，只优化现有功能的可达性、清晰度、一致性和稳定感**

## 一句话判断

这个项目的功能已经足够丰富，当前最影响体验的不是“功能少”，而是：

1. 首页和主工作台承载的信息过多，用户第一眼不知道“现在最该做什么”。
2. Web 和小程序两端的数据加载策略不完全一致，部分页面会出现“入口不同，体验不同”的问题。
3. 一些加载失败被静默吞掉了，用户看到的是空白、没数据、或者不确定是不是自己的操作有问题。

如果只做体验优化，我建议优先收敛这三件事。它们不会改变产品边界，但会明显提升“顺手程度”和“完成度”。

## 目前做得好的地方

先说结论：这不是一个“基础很差”的项目，反而已经有不少不错的体验底子，值得保留并扩展。

- Web 学生端已经做了按模块懒加载，避免首次把全部数据一次性打满。
  - 参考：`public/student-core.js`
- 小程序已经有骨架屏、bootstrap 缓存、下拉刷新这些基础体验能力。
  - 参考：`miniprogram/app.js`、`miniprogram/pages/home/index.wxml`
- Web 端已经有统一的 `toast` 和空状态组件，说明你们已经有“反馈一致性”的意识。
  - 参考：`public/common.js`
- 学生端 Web 已经开始拆分脚本，说明代码结构正在往可维护方向走。
  - 参考：`public/student.html` 末尾脚本引用，`public/student-core.js`、`public/student-courses.js` 等

也就是说，这个项目现在更适合做“收束和打磨”，而不是继续堆功能。

## 优先级建议

## P0：先做这 4 件，收益最大

### 1. 把学生端和教师端的首页重新聚焦成“一个主任务 + 少量次要入口”

#### 现状

学生端 Web 首页同时承载了任务、题库、课程、直播、学习数据、论坛、商城、词汇、打卡、成就、每日推荐等大量模块。

- 参考：`public/student.html:55-69`

教师端也是类似模式：任务规划、学生管理、课程、直播，再加上总结、论坛、题库、商城、词汇等入口都塞在同一工作台。

- 参考：`public/teacher.html:36-49`

这类结构的问题不是功能多，而是**首屏决策成本太高**。用户一登录就要先判断“我该去哪一块”，而不是直接进入当前目标。

#### 建议

- 学生端首页只保留 3 块：
  - 今日任务
  - 下一步动作
  - 最近继续内容（课程 / 刷题 / 错题复习三选一）
- 其余模块不要消失，但从“首页主内容”降级为“二级入口”。
  - 更适合独立页面或明确分组，例如“学习”“复盘”“服务”“我的”
- 教师端首页也做同样处理：
  - 今日待处理事项
  - 学生风险提醒
  - 高频操作入口
- 创建任务、导入任务、手动提醒这三类操作不要并排堆在一个长页面里，建议按“日常操作”和“批量操作”拆区。

#### 预期效果

- 首屏理解成本下降
- 用户更容易形成固定操作路径
- 每次登录后更快进入工作状态

---

### 2. 统一小程序的数据加载协议，消除“看入口吃缓存”的体验不确定性

#### 现状

服务端已经明确区分了“核心 bootstrap 数据”和“模块化数据”。

- 参考：`src/server.js:555-619`

但小程序端的 `fetchBootstrap()` 目前只拿核心 bootstrap，并没有模块参数能力。

- 参考：`miniprogram/app.js:27-88`

与此同时，课程页、课程详情页、商城页、订单页又直接从 `bootstrap` 里读取 `courses`、`products`、`orders`。

- 参考：`miniprogram/pages/courses/index.js:24-38`
- 参考：`miniprogram/pages/courses/detail.js:33-46`
- 参考：`miniprogram/pages/store/index.js:39-76`

这意味着这些页面的体验可能依赖“之前有没有从某个入口把数据灌进缓存”。从用户视角看，就会表现成：

- 有时候页面有数据
- 有时候空白
- 有时候提示不存在
- 但用户并不知道差别来自缓存状态

这类问题很伤“产品可信度”。

#### 建议

- 小程序端也改成和 Web 一样的模块化加载思路。
- `app.fetchBootstrap()` 保留核心数据，但新增 `fetchBootstrapModules(['courses'])` 这类能力。
- 各页面对自己依赖的数据负责：
  - 课程页主动加载课程模块
  - 商城页主动加载商品 / 订单模块
  - 详情页允许直接按 `id` 请求单条数据，不能只靠列表缓存反查
- 本地缓存继续保留，但只作为“加速手段”，不要成为“页面能否正常工作”的前提。

#### 预期效果

- 小程序页面入口一致性提升
- 减少“偶现空白”和“明明有数据却找不到”的体验问题
- 后续排查问题会容易很多

---

### 3. 把静默失败改成可见失败，避免用户误以为“系统没反应”

#### 现状

代码里有不少请求失败被直接吞掉的地方。

- 参考：`public/student-core.js:52-58`
- 参考：`public/teacher.js:45-53`
- 参考：`miniprogram/pages/questions/index.js:51-58`
- 参考：`miniprogram/pages/search/index.js:51-55`

静默失败对开发者友好，对用户不友好。因为用户不会知道这是：

- 网络问题
- 权限问题
- 数据为空
- 还是页面真的没有内容

#### 建议

- 对“关键数据区”统一区分 4 种状态：
  - 加载中
  - 加载失败
  - 空数据
  - 正常展示
- `catch` 里不要默认静默，至少做一层轻提示或内联错误提示。
- 对二级模块加一个统一的“重试”入口，而不是让用户靠刷新整个页面碰运气。
- 对失败频率高的模块，把错误文案做成用户语言，不要只是技术错误透传。

#### 预期效果

- 用户知道发生了什么
- 降低“系统是不是坏了”的不确定感
- 客服和排障成本下降

---

### 4. 收敛“单页过长工作流”，减少在同一屏里做太多事情

#### 现状

学生端“今日任务”区域下方直接接“今日总结提交”，再往后切别的模块；教师端“任务规划”区域里同时包含创建、导入、提醒、任务列表。

- 参考：`public/student.html:74-105`
- 参考：`public/teacher.html:53-120`

这会导致一种常见体验问题：**页面不是不好用，而是用户心理上觉得“重”**。

#### 建议

- 学生端把“做任务”和“提总结”分成两个明显阶段。
  - 上午和日间关注执行
  - 晚上关注复盘
- 教师端把“创建单条任务”和“批量导入/统一提醒”拆成两个工作区，不要默认都展开。
- 对低频区域默认折叠，但折叠后要保留清晰标题和结果摘要。

#### 预期效果

- 页面压力感下降
- 操作更符合真实使用节奏
- 学生和老师都更容易进入“顺手”的日常路径

## P1：第二阶段做，能显著提升完整度

### 5. 打通搜索结果和目标页，避免“搜到了，但点过去没有上下文”

#### 现状

小程序搜索页已经有搜索历史、热词和结果分类，这很好。但“题目结果跳转”目前没有把筛选上下文带过去。

- 参考：`miniprogram/pages/search/index.js:107-111`

也就是说，用户搜到题目后进入题库，可能还要重新找。

#### 建议

- 搜索跳转时把 `subject`、`mode`、`questionId` 这类上下文带过去。
- 目标页进入后优先定位结果，而不是只切到 tab。
- 课程、题目、帖子三类结果都应该尽量做到“点进去就是答案现场”。

#### 预期效果

- 搜索闭环完整
- 用户不会觉得“搜索只是看看，不是真正可达”

---

### 6. 统一反馈语言和状态语气，让整个平台显得更专业

#### 现状

现在 Web 端已经有很多 `toast`，这是好事，但不同模块里的反馈风格并不完全一致。有的是动作结果，有的是错误直出，有的是空状态提示。

#### 建议

- 统一 4 类文案模板：
  - 成功
  - 失败
  - 空状态
  - 引导下一步
- 优先说结果，再说原因，再说下一步。
  - 例如：“保存失败，请检查网络后重试”
  - 比单纯“请求失败”更好
- 管理端、教师端、学生端的语气可以略有差异，但句式和反馈节奏应一致。

#### 预期效果

- 整体专业感更强
- 用户更容易建立“系统会告诉我发生了什么”的预期

---

### 7. 给所有重操作增加“处理中”感，而不是只在完成后提示

#### 现状

项目里已经有按钮 loading 能力和骨架屏，但覆盖不够均匀。

- 参考：`public/common.js`
- 参考：`miniprogram/app.wxss`

有些操作能感知到正在处理，有些只能等结果。

#### 建议

- 对以下动作统一加处理中状态：
  - 提交总结
  - 上传文件
  - 下单
  - 导入 Excel
  - 创建直播
  - 生成试卷 / 加载错题 / 加载课程
- Web 端重点补 tab 首次加载的骨架或占位。
- 小程序端重点补长列表分页和二次进入的“轻加载”反馈。

#### 预期效果

- 用户不容易重复点击
- 系统响应体感更稳定
- 大操作不再像“卡住”

---

### 8. 强化“继续上次内容”的恢复能力

#### 现状

项目里已经有课程进度、最近课程、bootstrap 缓存、练习记录，这些都很好，但入口上的恢复感还可以再往前一步。

#### 建议

- 学生端首页把“上次学到哪”做成明确入口，而不是散落在课程、题库、词汇各自页面里。
- 教师端首页把“今天最值得处理的未完成事项”前置。
- 小程序首页保留当前结构，但把“下一步动作”卡片再明确成一个主 CTA。

#### 预期效果

- 打开即继续，不需要重新判断
- 学习产品常见的“断点感”会明显变弱

## P2：第三阶段做，主要提升质感和后续迭代效率

### 9. 统一 Web 和小程序的视觉语义，不要只统一颜色

#### 现状

Web 端偏控制台式工作台，小程序端更偏卡片流，两边的组件语言已经有些分叉。

#### 建议

- 统一状态色语义：
  - 主操作
  - 成功
  - 警示
  - 危险
- 统一常见控件的层级关系：
  - 主按钮
  - 次按钮
  - 文字按钮
  - 标签 / 徽章
- 统一空状态的结构：
  - 图形
  - 标题
  - 一句解释
  - 一个推荐动作

#### 预期效果

- 两端更像一个产品，而不是两个风格接近的项目
- 后续设计决策更稳定

---

### 10. 清理遗留前端实现，降低后续体验优化的维护成本

#### 现状

Web 学生端现在实际走的是拆分脚本：

- `public/student-core.js`
- `public/student-courses.js`
- `public/student-practice.js`
- `public/student-store.js`
- `public/student-social.js`
- `public/student-flashcards.js`
- `public/student-stats.js`

但仓库里仍保留了一份未被页面引用的 `public/student.js`，规模约 2748 行。

这个问题不是立刻影响用户，而是会影响你后面每次改体验时的判断成本：到底哪个版本是真实生效的？

#### 建议

- 明确保留哪一套实现
- 把未引用的旧入口归档或删除
- README 同步更新，不要继续写 `student.html / student.js`

#### 预期效果

- 迭代体验优化时不容易改错文件
- 降低“代码和实际运行不一致”的认知负担

### 11. 建一套“体验回归清单”，以后每次上线都过一遍

这个项目功能面已经比较大了，再继续靠肉眼检查会越来越吃力。建议做一个很轻量的体验回归清单，不需要很重。

#### 推荐至少覆盖

- 登录后 10 秒内能否知道下一步做什么
- 核心页面是否都有加载中 / 空状态 / 失败态
- 从搜索结果到目标内容是否可直达
- 小程序页面是否可从任意入口稳定进入
- 提交、上传、下单、导入时是否能看见处理中状态
- 断网、超时、401 时是否有明确反馈

#### 预期效果

- 每次上线都能稳住基础体验
- 体验问题从“上线后发现”转为“上线前发现”

## 建议的执行顺序

如果你准备真的动手做，我建议按下面顺序：

1. 先处理小程序数据加载协议
   - 这是稳定性问题，优先级最高
2. 再重做学生端 / 教师端首页的信息层级
   - 这是感知最明显的体验收益
3. 再统一失败态、空状态、处理中状态
   - 这是“完成度”的关键
4. 最后做搜索闭环、视觉统一和遗留代码清理
   - 这是提质增效

## 最后结论

这个项目已经不是“缺功能”的阶段了，而是到了一个很典型的节点：

**功能够多，但需要通过结构收敛、链路一致和反馈统一，让用户感觉它更稳、更顺、更像成品。**

如果只让我给一句最实用的建议，那就是：

> 先不要再想加什么，先把“用户打开后 3 秒内知道做什么、点了以后知道系统在做什么、从任意入口进入都能稳定拿到数据”这三件事做扎实。

只做这三件，整体体验就会明显提升一个档次。
