# 项目体验优化建议

> 评估时间：2026-04-17  
> 评估范围：Web 学生端、Web 教师端、微信小程序  
> 前提：**不新增功能，只优化现有功能的排布、反馈、一致性和完成度**

## 结论先说

这个项目现在的主要问题已经不是“功能不够”，而是“功能入口太平、工作流太长、状态反馈不够统一”。

从代码现状看，项目已经有不错的基础：

- Web 端已经有统一的卡片、按钮、Toast、空状态基础设施，见 `public/styles.css`、`public/common.js`
- 小程序已经有品牌色、骨架屏、按模块拉取 bootstrap 的能力，见 `miniprogram/app.js`
- 小程序首页已经开始围绕“今日规划”组织，而不是纯入口堆叠，见 `miniprogram/pages/home/index.wxml`

但真正影响体感的点主要集中在四类：

1. 首屏入口仍然偏多，用户第一眼不够容易知道“现在先做什么”
2. 同类页面的加载中 / 失败 / 空状态 / 处理中反馈不统一
3. Web 与小程序虽然共用同一套业务，但“工作台心智”还不够一致
4. 少数页面已经出现“完成度断点”，会直接拉低产品可信感

如果只做体验优化，我建议优先处理这些问题，而不是继续往里加模块。

---

## 当前观察

### 1. 小程序首页方向是对的，但还可以再收一层

`miniprogram/pages/home/index.wxml` 当前结构是：

- 第一屏：Hero + 核心指标 + 搜索
- 第二屏：下一步行动
- 第三屏：今日任务
- 第四屏：今日总结
- 第五屏：更多信息

这个方向明显比“模块总入口页”更好，但“下一步行动”里现在有 6 张权重几乎相同的卡片：

- 完成今日任务
- 开始刷题
- 继续课程
- 词汇复习
- 提交今日总结
- 论坛交流

问题不在于它们不能存在，而在于它们同时以同样的视觉重量出现，会把“主动作”稀释掉。用户进入首页之后，仍然需要自己判断优先级。

### 2. Web 学生端还是偏“大工作台”，而不是“主路径工作区”

`public/student.html` 当前把很多模块都挂在同一层：

- 今日任务
- 学习总结
- 题库刷题
- 录播课程
- 直播答疑
- 学习数据
- 更多里的论坛、商城、词汇、打卡、成就、每日推荐

这还不止。题库面板内部又有：

- 刷题模式切换
- 科目 / 题型 / 标签筛选
- 全部 / 错题 / 收藏二级切换
- 侧边答题卡

这会产生两个明显问题：

- 认知层级太深，用户切到一个面板后还要继续切
- 一个页面承载过多工作流，导致“不是不能用，而是心理上显得重”

### 3. Web 教师端任务规划区把“高频”和“低频”操作混在了一起

`public/teacher.html` 的“考研任务规划”面板里，当前同时存在：

- 单条任务创建
- 高级设置
- 批量导入
- 手动触发提醒
- 任务列表

这类排布对首次看系统的人很不友好，因为同一屏里既有“天天要做的事”，又有“偶尔才做的事”，用户会默认把整个区域理解成复杂、需要谨慎操作。

### 4. 状态反馈有基础，但不够彻底

项目里已经有一些不错的体验基础：

- Web 端有统一 `toast`、`empty-state`、`retry-state`，见 `public/common.js`
- 小程序课程页、商城页已经开始显式区分 `loading / loadError / empty`，见 `miniprogram/pages/courses/index.js`、`miniprogram/pages/store/index.js`

但仍有几类典型不一致：

- `miniprogram/pages/profile/index.js` 的 `loadDashboard()` 失败后直接静默
- `miniprogram/pages/questions/index.js` 的题目元数据、统计数据失败后只 `console.warn`
- 小程序首页很多失败只用 `toast`，但没有区块级错误态

结果就是：有的页面“像产品”，有的页面“像接口调失败后还在硬撑”。

### 5. 有一个现成的完成度断点需要优先修

`miniprogram/pages/courses/index.wxml` 当前课程列表容器存在，但没有渲染课程卡片内容。  
对应文件的 `.course-grid` 样式和 `.course-card` 样式都已经写好了，见 `miniprogram/pages/courses/index.wxss`，但 WXML 列表部分是空的。

这不是“优化建议”，而是应该尽快修掉的现有断点。它会直接让用户怀疑“课程是不是根本没做好”。

## 优先级建议

## P0：先做这些，收益最大

### 1. 把首页从“入口汇总页”收成“当前任务页”

这条建议适用于小程序首页，也适用于 Web 学生端 / 教师端首屏。

建议原则：

- 首屏只放一个主目标
- 次级入口可以保留，但不要和主目标同权
- 低频内容默认折叠或后置

#### 小程序首页建议排布

建议从现在的结构收成下面这个顺序：

```text
Hero（今日状态）
主 CTA：继续当前最重要的事
次级入口：刷题 / 课程 / 词汇
今日任务
今日总结
更多信息（折叠）
```

具体建议：

- “下一步行动”保留，但改成 `1 个主卡 + 3 个次卡`
- “论坛交流”不必出现在首页主操作区，放到 tab 或“更多信息”里就够了
- “提交今日总结”更适合在今日任务区后面强化，而不是与“开始刷题”并列
- 现在 6 张不同底色的小卡有点碎，建议只保留 2 到 3 种语义色，而不是每张卡都换一套浅色背景

#### Web 学生端建议排布

建议首屏只保留 4 个主工作区：

- 今日任务
- 学习总结
- 题库刷题
- 录播课程

其余内容：

- 直播答疑保留，但不要放在首层核心区
- 学习数据建议从“工作台入口”降为“结果页”
- 商城、论坛、词汇、打卡、成就、每日推荐都更适合归入次级导航

#### Web 教师端建议排布

建议首屏只保留 4 个主工作区：

- 今日待处理任务
- 学生风险与异常
- 课程运营
- 直播答疑

“题库管理 / 商城 / 论坛 / 词汇管理”不需要消失，但不应该跟任务规划同一层竞争注意力。

---

### 2. 拆掉“一个面板做完所有事”的排布

当前最典型的是教师端任务规划区。

建议把它拆成两个层级：

- 高频区：创建任务、查看今日任务、查看未完成任务
- 低频区：批量导入、手动提醒、复杂高级设置

更具体一点：

- 单条任务创建保留在默认展开区
- 批量导入和手动提醒放入“批量与运营操作”折叠区
- 高级设置不要默认参与用户首屏决策
- 任务列表不要紧跟在一大段表单之后，建议和“创建区”形成明显分隔

学生端题库区也有类似问题：

- 当前在一个面板里叠了模式切换、筛选、题目列表、分页、报告、侧边答题卡
- 建议至少让“做题区”比“筛选区”更像主角
- 筛选只在展开时出现，进题以后应尽量让用户聚焦题目本身

---

### 3. 统一四种状态：加载中 / 加载失败 / 空状态 / 处理中

这条建议不需要加新功能，只要把已有组件和模式用一致。

建议最低统一标准：

- 列表页必须有骨架屏或占位
- 核心区块必须有失败态和重试入口
- 没数据时给出明确空状态，不要只剩白屏
- 提交 / 上传 / 下单 / 导入时，必须有“处理中”反馈，不要只在成功后提示

优先统一的页面：

- 小程序首页
- 小程序个人中心
- 小程序题库
- Web 学生端题库 / 商城
- Web 教师端任务导入 / 直播创建 / 文件上传

特别说明：

- `console.warn` 对开发者有用，对用户没有用
- 静默降级只能用于次要信息，不能用于主内容区

---

### 4. 收敛视觉语义，不要让每块卡片都像不同页面

当前 Web 和小程序都已经有蓝色品牌主色，这是好事。问题在于局部组件还是有很多“各写各的”的痕迹：

- 大量内联样式
- 同类卡片使用不同圆角、间距、字号
- 首页操作卡过度依赖不同浅色底
- 许多空状态、按钮、徽标的语气不完全一致

建议：

- 只保留 4 个语义色：主操作、成功、警示、危险
- 优先用类名和变量表达语义，减少散落在 WXML / HTML 中的内联样式
- 首页操作卡的颜色不要一张一个主题
- Emoji 可以继续保留，但建议减少到“辅助点缀”而不是“主要识别”

这一步的价值不只是更好看，更重要的是后续改版会轻很多。

---

### 5. 先修现有断点，再谈“高级体验”

有几处已经属于完成度问题，建议在所有“体验优化”之前优先修：

- `miniprogram/pages/courses/index.wxml`：课程列表未渲染
- `miniprogram/pages/profile/index.wxml`：没有显式 loading / error 结构，但 `index.js` 里存在加载流程
- `public/student.html` 题库区层级较深，窄屏时会更容易显得拥挤

体验优化最怕“视觉建议讲了很多，但用户一进页面先看到一个没渲染完的模块”。

## P1：第二阶段做，能明显提升完整度

### 6. 让 Web 与小程序的“主路径”保持一致

现在两端都服务同一批业务，但主路径表达还不够统一。

建议统一成这条产品心智：

```text
看今天要做什么
开始执行
中途回到上次进度
结束后复盘
最后看统计结果
```

如果两端都围绕这条主路径组织，用户切换设备时会更顺。

### 7. 个人中心从“信息堆叠页”改成“结果页”

`miniprogram/pages/profile/index.wxml` 当前内容不少：

- Hero 统计
- 快捷入口
- Tab
- 学习日历
- 科目正确率
- 薄弱点
- 成就
- 记录
- 账号

建议把它的角色收窄成“结果与回顾页”，而不是“另一个首页”：

- 快捷入口保留 2 个即可，不必再复制首页导航能力
- 学习数据和记录优先
- 成就和账号放后面
- 不要让个人中心承担过多“继续学习”的入口职责

### 8. 搜索页更强调“结果浏览”，弱化“功能切换感”

`miniprogram/pages/search/index.wxml` 当前是典型 Tab 搜索页，结构没问题，但还可以更像“内容结果页”：

- 结果卡片之间的视觉层级再清楚一点
- “全部 / 帖子 / 题目 / 课程”切换可以更轻，不必和主 tab 一样重
- 搜索结果的元信息可以更集中，不要让每张卡像独立页面

这一步不需要加新能力，主要是把视觉密度收紧。

### 9. Web 响应式不要只做“一列化”，还要做“层级减法”

`public/styles.css` 已经有基础断点，但现在更多是布局塌成单列，并没有同步减少页面复杂度。

建议：

- 窄屏下减少首层 tab 数量
- 把复杂筛选改成折叠面板
- 让次级功能延后显示
- 取消不必要的并列区域

真正的移动端友好，不是把原页面压窄，而是把决策数变少。

## 前端排布建议

下面是我更推荐的排布方式，都是基于现有功能重排，不新增模块。

### 1. 小程序首页

```text
[Hero]
今天状态 / 未完成数 / 最近直播 / 搜索

[主动作]
继续当前最重要的事

[次动作]
刷题 / 课程 / 词汇

[今日任务]
默认展开

[今日总结]
默认显示，但视觉弱于任务区

[更多信息]
提醒 / 直播 / 历史总结，默认折叠
```

关键点：

- 首页只负责“开工”和“收尾”
- 论坛、商城、社区不是首页主职责
- 任务区永远比入口区更重要

### 2. Web 学生端

```text
左侧
- 账号
- 提醒中心
- 轻量设置

主区第一屏
- 今日状态
- 下一步

主区第二层
- 今日任务
- 学习总结
- 题库
- 课程

主区第三层
- 数据 / 直播 / 论坛 / 商城 / 词汇 等次级内容
```

关键点：

- 工作流优先，不是模块优先
- 尽量避免“tab 里再套一层 tab”
- 题库和课程这种高使用频场景，应该拿到更完整的主区空间

### 3. Web 教师端

```text
左侧
- 账号
- 简要说明

主区第一屏
- 今日概览
- 风险提醒

主区第二层
- 快捷操作：创建任务 / 导入 / 提醒

主区第三层
- 任务列表
- 学生表现

主区第四层
- 课程 / 直播 / 题库 / 论坛 / 商城 / 词汇
```

关键点：

- 高频操作前置
- 运营类、批量类动作后置
- 列表和表单不要互相打断

## 可以直接执行的落地顺序

如果只做体验优化，我建议按下面顺序推进：

1. 先修现有断点  
   先把课程页这种“完成度问题”修掉，再谈视觉体验

2. 再重排首页和工作台  
   这是用户最容易感知到的提升

3. 再统一状态反馈  
   把 loading / error / empty / processing 全部收口

4. 最后做视觉收敛  
   包括颜色语义、内联样式清理、组件样式统一

---

## 最后判断

这个项目现在已经过了“继续堆功能”的阶段，更适合做一次“收口”。

真正能提升体验的，不是再加两个模块，而是把现有模块重新排成一条更顺的主路径：

- 用户一进来知道先做什么
- 做的过程中知道系统正在处理什么
- 出错时知道发生了什么
- 结束后知道下一步去哪里

只要把这四件事做好，整个项目的“专业感”和“完成度”会明显上一个台阶。
